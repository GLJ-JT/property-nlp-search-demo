The problem

The search session breaks before the shortlist forms.

Property portals already do structured search well. The gap appears when renters arrive with mixed intent: budget, commute, school, station, building type, exclusions, and vague language in one sentence.

The core issue is not awareness or traffic. The issue is trust at the moment of intent. If the product cannot show that it understands the query, the renter starts comparing outside the portal.

That leakage often happens through WhatsApp, WeChat, Telegram, screenshots, agent calls, and notes apps. The portal may still receive visits, but it stops owning the decision process.

The opportunity is to make the first messy query feel understood enough that the renter keeps refining inside the product.

The renter does not need a magic search box. They need proof that the system understood the tradeoffs.
01 Current

Query is compressed into filters

A renter types something like "studio near UCL under 2500 bills included lift quiet". The product may handle budget and location, but the combined intent is still mostly hidden.

02 Friction

False positives reduce trust

If basement flats, distant listings, or wrong room types appear, the renter has to police the product. That is where confidence drops.

03 Leakage

Comparison moves off-platform

The user screenshots, asks agents, checks social channels, and builds the shortlist somewhere else. The portal becomes one input, not the operating system for the search.

The proposal

Add an intent layer, not a replacement search engine.

The strongest version of this idea is conservative: preserve existing filters, then add natural-language parsing, editable chips, and explainable ranking on top.

Listen

Accept messy renter language

Let renters type constraints in the order they think of them. The system reads budget, room type, landmarks, station distance, amenities, exclusions, and soft preferences.

Show

Return intent as editable chips

Every parsed constraint becomes visible. Users can remove or edit chips, so the product feels transparent instead of magical.

Rank

Separate hard filters from nice-to-haves

Must-haves remove bad matches. Preferences reorder good matches. This distinction is the difference between helpful NLP and a black box.

Explain

Make ranking reasons visible

Each result explains why it matched: near a landmark, inside budget, correct room type, lift, bills included, or a weaker soft fit.

Is it strong enough?

Yes, if you keep the claim precise.

The generic version is "AI search for property". The strong version is "visible intent capture that reduces early leakage and increases trust before shortlist formation".

What wins interview progress

The proposal connects a business problem to a user problem, then proves execution through a working prototype. That is much stronger than a speculative AI concept.

What could sound generic

Claims about revenue uplift, personalization, or "AI understands renters" become weak if they are not tied to a concrete mechanism.

How to say it

"I am not proposing replacing search. I am proposing an explainable intent layer that helps the existing search system keep high-intent renters inside the product longer."

Next 2 weeks: validation

Test 20 real renter queries, compare parsed intent against manual filters, and identify where the product should ask clarifying questions.

Next 6 weeks: controlled pilot

Run the layer behind a feature flag for saved-search users or relocation-heavy segments. Measure query reformulation, saves, contact clicks, and session continuation.

Next 12 weeks: production decision

Decide whether the value is strongest in search input, saved alerts, agent lead quality, or onboarding for new renters.
