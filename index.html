What this is

A mobile-first property search concept for high-intent renters.

The prototype explores how a property portal could support messy renter language without replacing its existing search model.

Property search is already crowded with filters, maps, agent copy, saved alerts, and paid acquisition journeys. The problem is not that users lack ways to search. The problem is that high-intent users often arrive with mixed requirements that do not fit neatly into one control.

They might care about a landmark, budget, station distance, bills, floor level, accessibility, room type, and exclusions in the same breath. If the product cannot hold that combined intent, users start creating their own comparison system outside the platform.

My work focused on the product experience layer behind that gap: the natural-language entry point, editable intent chips, ranking explanations, clarification prompts, and the reusable data structure needed to support the same behaviour across mobile and web.

Mobile-led expectation

The small-screen journey has to support fast understanding. Users should not have to open every filter drawer before the system can help.

Search complexity

Property listings combine structured facts and agent-written text. Ranking needs to separate verified facts from derived semantic signals.

Trust pressure

False positives damage confidence quickly. A single basement flat after "not basement" teaches the user to leave.

The problem was a confidence leak

The user journey leaked before the shortlist existed.

The friction was not isolated to one screen. It appeared across a connected search journey: query entry, filter interpretation, result ranking, listing comparison, clarification, and save/share behaviour.

The early product signal was behavioural. Renters could start with a strong intent, but the interface did not always reflect that intent back clearly enough. When users could not tell why a listing appeared, or whether the system had respected an exclusion, they had to police the results manually.

That is where the proposition becomes commercially relevant. Paid traffic can create sessions, but trust in the search experience determines whether those sessions become saved properties, enquiries, return visits, or off-platform comparison.

Reading guide: mobile was the main stage; natural-language entry, intent chips, and explainable ranking were the friction points; repeated uncertainty became repeated leakage.

Diagnosis: from screens to patterns

The issue was not search input. It was search confidence.

I looked at how a renter's intent should survive the journey from query to result card. The key was to make the system's interpretation visible and editable.

Observed behaviour

Users express constraints together: "studio near UCL under 2500 bills included lift quiet". A standard filter-first flow forces them to break that thought apart too early.

User expectation

Renters expect the product to understand places, stations, budgets, exclusions, and soft preferences as one search task, then let them adjust the interpretation.

System decision

The search became a progressive intent pattern: parse the query, show editable chips, ask clarifying questions when needed, and explain why each result ranked.

Reusable rule

High-intent search actions should keep the user's original language, parsed constraints, confidence prompts, and ranking reasons in the same decision zone.

System redesign

The prototype turns renter language into product infrastructure.

The design was not handed over as a static UI refresh. It became a working system with frontend states, backend endpoints, JSON data, ranking rules, and a whiteboard view for explanation.

Natural-language query

The search input was treated as a high-intent product surface. It accepts messy renter language, keeps the original query visible, and supports reruns after the user edits chips or chooses a clarification.

This matters because users should not lose the language they arrived with. The product can add structure, but it should not erase the user's mental model.

Ranking and explanation

The backend separates hard filters from soft preferences. Hard constraints remove bad matches. Soft preferences change order. Exclusions are treated as trust-critical rules.

Each result carries reasons such as budget fit, landmark proximity, room type, lift, bills included, semantic fit, or vector soft fit. The result is not just ranked. It is explainable.

State language

The search system needed explicit states, not vague AI magic.

Like a commerce buybox, the NLP layer needs reusable states so the interface can respond clearly at each decision point.

Search homes Default search

Used when the query is valid and the system can run the parser.

Choose UCL East Clarification

Appears when the backend needs a user choice before confidence improves.

Running Loading

Confirms the backend is parsing and ranking so users do not repeat-tap.

No strong matches Recovery

Shows how to relax constraints instead of leaving the user at a dead end.

What would mature at enterprise scale

From prototype to governed search system.

At scale, this would need stronger governance around data quality, accessibility, model confidence, component states, and experimentation.

Search token taxonomy

Define reusable tokens for place, commute, budget, amenities, exclusions, accessibility, and confidence level.

Component state matrix

Document query, chip, clarification, loading, empty, error, result, and relaxed-match states across mobile and desktop.

Accessibility and trust

Clarify keyboard behaviour, focus order, screen-reader labels, contrast, and error recovery for search and filter changes.

Experimentation workflow

Validate with real renter queries, compare against manual filters, and measure saves, contact clicks, reformulation, and return sessions.

Reflection

Design systems start where users repeatedly pause, doubt, or abandon.

In this project, the repeated hesitation was not a visual inconsistency. It was an intent inconsistency. Renters knew what they wanted, but the product needed a clearer way to hold that intent, expose it, and let them correct it. By turning that repeated hesitation into a reusable search pattern, the system becomes more than an interface improvement. It becomes shared product language for how search should understand, explain, and recover.
